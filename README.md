# vibe-meetings

A native macOS desktop app that records, transcribes, and summarises meetings
**100% locally**. No data ever leaves your Mac.

- 🎙 **System audio + microphone** captured as two parallel streams, so the
  transcript is automatically labelled `You` vs `Others`.
- 📝 **Reliable transcripts first.** Pluggable engines: WhisperKit (default,
  Apple Silicon optimised) and whisper.cpp (scaffolded — see notes below).
  Default model is `whisper-medium` (~1.5 GB, downloaded on first run).
- 🧠 **Local LLM summaries** via your own [Ollama](https://ollama.com)
  instance on `127.0.0.1:11434`.
- 🗂 **Filesystem-native storage.** Each meeting is a folder under
  `~/MeetingNotes/` containing `meeting.json`, `transcript.md`, `summary.md`,
  `segments.json`, and (optionally) `audio.m4a` (mic on L, system on R).
- 🔁 **Re-transcribe later** with a different engine/model — the audio is kept
  by default so you can always go back.
- 📥 **Audio file import** for transcribing existing recordings.

## Getting started (on a Mac)

### Prerequisites

- macOS 15 Sequoia or newer
- Xcode 16
- [XcodeGen](https://github.com/yonaskolb/XcodeGen) — `brew install xcodegen`
- [Ollama](https://ollama.com) (optional but needed for summaries) —
  `brew install ollama` then `ollama serve`

### First-run

```bash
git clone <this repo>
cd vibe-meetings
xcodegen generate          # creates VibeMeetings.xcodeproj from project.yml
open VibeMeetings.xcodeproj
```

In Xcode:

1. Pick the `VibeMeetings` scheme, select your Mac as the run destination.
2. Signing → "Sign to Run Locally" is fine for personal use.
3. Hit Run. The first launch will:
   - Prompt for **microphone access** (`NSMicrophoneUsageDescription`).
   - Prompt for **system audio access** the first time you start a recording
     (`NSAudioCaptureUsageDescription`). If you deny this, only your mic is
     transcribed.
4. Open Settings → Engines and choose your transcription model. Click
   "Download" next to `medium` (the default, ~1.5 GB). Models are cached at
   `~/Library/Application Support/VibeMeetings/Models/`.
5. Pull an Ollama model for summaries:
   ```bash
   ollama pull llama3.1:8b-instruct-q4_K_M
   ```
6. Press ⌘N to start your first meeting.

### Folder structure

Each meeting is one folder under your chosen root (`~/MeetingNotes/` by default):

```
~/MeetingNotes/
└── Work/
    └── 2026-05-10 Standup/
        ├── meeting.json       # canonical metadata
        ├── transcript.md      # human-readable, Obsidian-friendly
        ├── summary.md         # LLM output
        ├── segments.json      # source of truth for re-renders
        └── audio.m4a          # 2-channel: mic on L, system on R
```

A folder is a "meeting" iff it contains `meeting.json`. Rename / move /
delete folders in Finder — the sidebar reflects the change within ~1 s
thanks to FSEvents, and the `meeting.id` UUID survives moves.

## Privacy posture

- App Sandbox is **off** (Core Audio Tap requires it). Hardened Runtime is on.
- The only network connections this app ever makes:
  1. `127.0.0.1:11434` — your local Ollama, only when summarizing.
  2. Hugging Face / argmax CDN — **only when you click "Download"** for a
     transcription model.
- All `URLSession` usage routes through `LocalhostOnlySession` which rejects
  any non-loopback host. The model downloader is the only deliberate
  exception.
- No telemetry. No analytics. No crash reporting SDK. No auto-update.

## What to verify on a real Mac (the parts that need device-level testing)

### Phase 4 — System audio (`SystemAudioCapturer.swift`)

The Core Audio Tap API requires a real audio session. The skeleton creates a
`CATapDescription`, builds an aggregate device, and installs an IOProc — but
the byte-level copy from `AudioBufferList` into `AVAudioPCMBuffer` is the
classic place where format assumptions bite. Verify by:

```bash
# 1. Run the app, start a recording, play known audio (e.g. a YouTube clip).
# 2. Stop. Inspect the saved m4a:
afinfo ~/MeetingNotes/Work/<meeting>/audio.m4a
# Expect: 2 channels, 48000 Hz, AAC.
# 3. Solo each channel in Audacity (File → Import → Audio):
#    - Left = your mic
#    - Right = the YouTube audio
```

If channels are swapped or one is silent, the most likely culprits are the
channel routing in `DualChannelM4AWriter.flushIfReady()` or the tap stream
format extracted in `SystemAudioCapturer.streamFormat`.

### Phase 5 — WhisperKit

The `WhisperKitEngine.swift` file uses the WhisperKit Swift API. WhisperKit's
public API has had small renames between 0.7 / 0.8 / 0.9 / 0.10 — if Xcode
shows compile errors after package resolution, check the changelog and update
the call sites in `transcribeFile` and `transcribeStream`. The pinned version
in `Packages/VMTranscription/Package.swift` is `from: "0.9.0"`; bump as needed.

### Phase 6 — whisper.cpp

Currently a stub that throws `engineUnavailable`. Follow the four-step
integration in
`Packages/VMTranscription/Sources/VMTranscription/WhisperCppBridge/README.md`.

## Verification checklist (end-to-end)

1. Build: `xcodegen generate && xcodebuild -project VibeMeetings.xcodeproj -scheme VibeMeetings -destination 'platform=macOS'`
2. Onboarding: grant mic + audio capture permissions, download `medium`,
   confirm Ollama health is green.
3. Create a folder `Work/`, hit ⌘N, start a meeting while playing a sample
   call audio.
4. Verify both VU meters move and live transcript shows interleaved
   `You` / `Others` lines.
5. Stop. Inspect the meeting folder — should contain `meeting.json`,
   `transcript.md`, `segments.json`, `audio.m4a` (2 channels confirmed via
   `afinfo`). `summary.md` is generated by Ollama.
6. Quit, rename the meeting folder in Finder, reopen the app — sidebar
   reflects the rename within ~1 s; the meeting still opens.
7. Privacy check: with **Little Snitch** or **LuLu** running, exercise every
   feature. Confirm zero network connections except `127.0.0.1:11434` and
   the explicit model download you initiated.
8. Tests: in each `Packages/VM*/` directory, `swift test` should pass once
   tests are added.
